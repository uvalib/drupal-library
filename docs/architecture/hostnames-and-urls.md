# Hostnames & URLs

A single place to find the browser URLs, host-header aliases, and backend/SSH
hosts for every environment — so you don't have to reassemble them from
`dns.tf` / `alb-routing.tf` / `output.tf` across three environment directories.

!!! warning "This is a convenience copy — keep it honest"
    The **authoritative source** is `terraform-infrastructure/library.virginia.edu/<env>/`
    (`dns.tf`, `alb-routing.tf`, `output.tf`) and the generated
    `<env>/ansible/inventory.generated` / `tfvars.*.generated`. This page is a
    hand-maintained snapshot; when the ALB rules, CNAMEs, or instance counts
    change in Terraform, update this page too. Last reconciled: **2026-07-01**.

## Browser URLs (the front door)

What you type into a browser for each environment.

| Environment | URL | Visibility |
|-------------|-----|------------|
| **Production** | `https://library.virginia.edu` (also `https://lib.virginia.edu`) | Public internet |
| Production (internal LB CNAME) | `https://library-drupal.internal.lib.virginia.edu` | UVA network / VPN |
| **Staging** | `https://library-drupal-staging.internal.lib.virginia.edu` | UVA-only (VPN required) |
| **Develop** | `https://library-drupal-develop.internal.lib.virginia.edu` | UVA-only (VPN required) |

HTTPS is terminated at the upstream load balancer; the containers serve HTTP on
port 80 (see [ADR 002](../adr/002-https-at-load-balancer.md)). Staging and develop
are `uvaonly` ALB visibility, so their URLs only resolve and respond on the UVA
network / VPN.

### DevOps docs

The internal operations/architecture docs (this site) are served in-app at
**`/devops-docs`** on every environment, gated by the `devops` Drupal role over
NetBadge — e.g. `https://library-drupal-staging.internal.lib.virginia.edu/devops-docs`.

## Production ALB host-header routing

The production ALB (`uva-alb-public-production`) routes by `Host:` header. All of
the following point at the same load balancer; the rules decide what happens:

| Host / path | Behaviour |
|-------------|-----------|
| `library.virginia.edu`, `lib.virginia.edu` | Serve the site (forward to both prod nodes) |
| `preview.library.virginia.edu` | **Default: 301 redirect → `library.virginia.edu`.** When `preview_passthrough_enabled = true` (a Terraform var), forwards to **prod node 0 only** — a way to view one specific backend node, bypassing load balancing. See the [preview redirect session log](../session-logs/2026-07-01-preview-hostname-redirect.md). |
| `www.lib.virginia.edu`, `www.library.virginia.edu` | 301 redirect → `library.virginia.edu` |
| `libra.virginia.edu`, `libraprod.lib.virginia.edu` | 301 redirect → `library.virginia.edu/libra/` |
| `library.virginia.edu/pressbooks` | 301 redirect → `pressbooks.library.virginia.edu` |

There is **no dedicated Route53 record** for `preview.library.virginia.edu` in
this Terraform config — the public DNS entry lives outside the repo and points at
the same production ALB.

## Backend / SSH hosts

The actual EC2 instances running the `drupal-0` container. All are on
`*.internal.lib.virginia.edu` and require the **UVA VPN** (names won't resolve
without it). SSH is key-based; over VPN a plain `ssh <host>` works.

| Environment | Host | IP | Nodes |
|-------------|------|-----|-------|
| **Staging** | `library-drupal-staging-0.internal.lib.virginia.edu` | `10.130.109.122` | 1 |
| **Develop** | `library-drupal-develop-0.internal.lib.virginia.edu` | `10.130.109.80` | 1 |
| **Production** node 0 | `library-drupal-0.internal.lib.virginia.edu` | `10.130.110.83` | 2 (behind ALB) |
| **Production** node 1 | `library-drupal-1.internal.lib.virginia.edu` | `10.130.110.179` | 2 (behind ALB) |

Containers inside each host: **`drupal-0`** (Drupal) and **`netbadge-0`** (the
SimpleSAMLphp SP). Run a command in the Drupal container with, e.g.:

```bash
ssh library-drupal-staging-0.internal.lib.virginia.edu \
  'sudo docker exec drupal-0 drush status'
```

**Production runs two nodes** (`instance_count = 2`) behind the ALB — when
verifying a prod deploy, check **both**. Staging and develop are single-node.

## Where the naming comes from

The hostnames are generated by Terraform, so they follow a predictable pattern:

- **Public/LB CNAME:** `library-drupal-<env>` (production drops the suffix → just
  `library-drupal`), in the `internal.lib.virginia.edu` zone. So staging's LB CNAME
  is `library-drupal-staging.internal.lib.virginia.edu`.
- **Backend nodes:** `library-drupal-<env>-<index>.internal.lib.virginia.edu`
  (production drops `<env>` → `library-drupal-0` / `library-drupal-1`).

To re-confirm any of this authoritatively:

```bash
# The browser/front-door URL for an env:
cd terraform-infrastructure/library.virginia.edu/<env> && terraform output public_endpoint

# The backend host/IP list:
cat terraform-infrastructure/library.virginia.edu/<env>/ansible/inventory.generated
```
